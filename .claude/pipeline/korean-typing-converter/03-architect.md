# 기술 설계: korean-typing-converter

## 확정 사항
- **slug**: `korean-typing-converter`
- **category**: `text`
- **template**: `TextToText`
- **processingType**: `client`
- **icon**: `Languages` (lucide-react)
  - 대안: `Keyboard`. 양방향 언어 변환 성격이 강해 `Languages`를 1순위로 선정.

## inputConfig

```ts
inputConfig: {
  placeholder: "영문을 한글로, 한글을 영문으로 자동 변환합니다. 예: dksdud → 안녕",
  inputLabel: "텍스트 입력 (한글 또는 영문)",
  outputLabel: "변환 결과",
  outputType: "stats",
}
```

- `outputType`은 `"stats"`를 사용한다. 이유: 변환 결과 외에도 "감지된 방향", "원본 길이", "변환 길이" 등 부가 정보를 함께 보여주면 사용자 체감 가치가 커지기 때문.
- 단일 텍스트 결과만 원할 경우 `"text"`로 변경 가능하나, TextToText 템플릿의 stats 레이아웃이 word-counter에서 이미 검증되어 있으므로 stats를 택한다.

## process 시그니처 및 반환 key

시그니처:
```ts
process(input: string): Record<string, string | number>
```

반환 key (한국어 기준, stats 카드 라벨로 그대로 노출됨):
- `"변환 결과"`: `string` — 변환된 텍스트 (메인 출력)
- `"감지된 방향"`: `string` — `"영타 → 한글"` | `"한글 → 영타"` | `"변환 불가"`
- `"원본 길이"`: `number` — `input.length`
- `"변환 길이"`: `number` — 결과 문자열 길이

방향 감지 규칙:
1. input을 순회하며 첫 번째로 등장하는 "한글 음절(AC00–D7A3) 또는 한글 자모(1100–11FF, 3130–318F)"을 찾으면 → 한→영 모드
2. 첫 번째로 등장하는 "영문 알파벳(A–Z, a–z)"을 찾으면 → 영→한 모드
3. 둘 다 없으면 `"변환 불가"`로 표시하고 input을 그대로 반환
4. 공백, 숫자, 구두점, 기타 문자는 방향 판단에서 무시하고, 변환 시에는 원본 그대로 통과(passthrough)

## Logic Engineer를 위한 계산 명세

### 1. 두벌식 ↔ QWERTY 키 매핑 테이블

두벌식 표준(KS X 5002) 기준. **소문자 영문 → 한글 자모**:

| key | jamo | key | jamo | key | jamo | key | jamo |
|-----|------|-----|------|-----|------|-----|------|
| q | ㅂ | w | ㅈ | e | ㄷ | r | ㄱ |
| t | ㅅ | y | ㅛ | u | ㅕ | i | ㅑ |
| o | ㅐ | p | ㅔ | a | ㅁ | s | ㄴ |
| d | ㅇ | f | ㄹ | g | ㅎ | h | ㅗ |
| j | ㅓ | k | ㅏ | l | ㅣ | z | ㅋ |
| x | ㅌ | c | ㅊ | v | ㅍ | b | ㅠ |
| n | ㅜ | m | ㅡ |   |   |   |   |

**대문자(Shift) → 한글 자모**:

| key | jamo | key | jamo | key | jamo | key | jamo | key | jamo |
|-----|------|-----|------|-----|------|-----|------|-----|------|
| Q | ㅃ | W | ㅉ | E | ㄸ | R | ㄲ | T | ㅆ |
| O | ㅒ | P | ㅖ |   |   |   |   |   |   |

나머지 대문자(A,S,D,F,G,H,J,K,L,Z,X,C,V,B,N,M,Y,U,I)는 **소문자와 동일한 자모**로 매핑한다(두벌식 표준 동작).

역매핑(한글 자모 → QWERTY)은 위 표를 뒤집어 Map 하나로 구성. shift 자모(ㅃ,ㅉ,ㄸ,ㄲ,ㅆ,ㅒ,ㅖ)는 대문자 키로, 나머지는 소문자 키로.

### 2. 한글 유니코드 공식

완성형 음절 블록: `AC00` ('가') ~ `D7A3` ('힣')

**분해 (음절 → 초/중/종 인덱스)**:
```
code = char.charCodeAt(0) - 0xAC00
cho  = Math.floor(code / 588)          // 0..18
jung = Math.floor((code % 588) / 28)   // 0..20
jong = code % 28                       // 0..27 (0 = 종성 없음)
```

**조합 (초/중/종 인덱스 → 음절)**:
```
char = String.fromCharCode(0xAC00 + cho * 588 + jung * 28 + jong)
```

### 3. 인덱스 테이블 (상수)

```ts
// 초성 19자 (인덱스 0..18)
const CHO = ["ㄱ","ㄲ","ㄴ","ㄷ","ㄸ","ㄹ","ㅁ","ㅂ","ㅃ","ㅅ",
             "ㅆ","ㅇ","ㅈ","ㅉ","ㅊ","ㅋ","ㅌ","ㅍ","ㅎ"];

// 중성 21자 (인덱스 0..20)
const JUNG = ["ㅏ","ㅐ","ㅑ","ㅒ","ㅓ","ㅔ","ㅕ","ㅖ","ㅗ","ㅘ",
              "ㅙ","ㅚ","ㅛ","ㅜ","ㅝ","ㅞ","ㅟ","ㅠ","ㅡ","ㅢ","ㅣ"];

// 종성 28자 (인덱스 0..27, 0은 "종성 없음")
const JONG = ["","ㄱ","ㄲ","ㄳ","ㄴ","ㄵ","ㄶ","ㄷ","ㄹ","ㄺ",
              "ㄻ","ㄼ","ㄽ","ㄾ","ㄿ","ㅀ","ㅁ","ㅂ","ㅄ","ㅅ",
              "ㅆ","ㅇ","ㅈ","ㅊ","ㅋ","ㅌ","ㅍ","ㅎ"];
```

### 4. 복합 모음(중성) 합성 규칙

두벌식은 두 개의 모음 키를 연달아 눌러 복합 중성을 만든다:
```
ㅗ + ㅏ = ㅘ
ㅗ + ㅐ = ㅙ
ㅗ + ㅣ = ㅚ
ㅜ + ㅓ = ㅝ
ㅜ + ㅔ = ㅞ
ㅜ + ㅣ = ㅟ
ㅡ + ㅣ = ㅢ
```

합성 불가능한 모음 시퀀스(예: ㅏ+ㅣ)는 합성하지 않고 이전 음절을 종료한 뒤 새 음절을 시작한다.

### 5. 복합 종성(겹받침) 합성 규칙

```
ㄱ + ㅅ = ㄳ
ㄴ + ㅈ = ㄵ
ㄴ + ㅎ = ㄶ
ㄹ + ㄱ = ㄺ
ㄹ + ㅁ = ㄻ
ㄹ + ㅂ = ㄼ
ㄹ + ㅅ = ㄽ
ㄹ + ㅌ = ㄾ
ㄹ + ㅍ = ㄿ
ㄹ + ㅎ = ㅀ
ㅂ + ㅅ = ㅄ
```

### 6. 영 → 한 자동 조합 상태머신 (핵심 알고리즘)

입력 문자열을 char 단위로 순회하며 현재 "조립 중인 음절" 버퍼를 유지한다. 상태:
- `cho`: 초성 인덱스 | null
- `jung`: 중성 인덱스 | null
- `jong`: 종성 인덱스 | null

각 영문 char에 대해:

1. 매핑 테이블에서 자모를 찾는다. 없으면(공백/숫자/기호 등) → 현재 버퍼를 flush하고 원본 char를 출력에 그대로 append.
2. 자모가 **자음**인 경우:
   - `cho === null` → `cho = 자음 인덱스`
   - `cho !== null && jung === null` → 이전 자모는 초성이 아니라 **단독 자음**이었음 → flush 후 새 초성으로 시작
   - `jung !== null && jong === null` → 이 자음을 **종성 후보**로 설정 (단, 종성에 존재하는 자음에 한정. 종성에 없는 자음(ㄸ,ㅃ,ㅉ)이면 flush 후 새 초성)
   - `jong !== null` →
     - 겹받침 합성이 가능한 조합이면 `jong`을 겹받침 인덱스로 갱신
     - 불가능하면: **종성을 떼어서 다음 음절의 초성으로 이동**. 즉 현재 jong을 제거하고 그 종성을 새 음절의 cho로 시작한 뒤, 이번 입력 자음을 다시 처리(실제로는 현재 음절을 cho/jung까지만 flush → 떼어낸 자음 cho로 새 버퍼 → 이번 입력 처리 로직 재실행)
     - 단, 현재 종성이 **겹받침**이고 이번 입력이 모음이면, 겹받침의 뒤쪽 한 글자만 떼어서 다음 초성으로 이동(한글 IME 동작)
3. 자모가 **모음**인 경우:
   - `cho === null` → 초성 없이 모음만 있는 경우. `ㅇ`을 초성으로 넣지 않고 모음 단독으로 출력하려면: 버퍼를 flush하고 해당 자모(모음)를 그대로 출력. (단순화 구현: 모음 단독은 호환 자모 문자 그대로 append)
   - `jung === null` → `jung = 모음 인덱스`
   - `jung !== null && jong === null` → 복합 모음 합성 시도. 가능하면 `jung` 갱신, 불가능하면 flush(현 음절 완성) 후 새 음절을 `cho = null, jung = 모음`으로 시작. (이 경우 이전 초성이 있었으므로 완성 음절 출력됨)
   - `jong !== null` → **종성을 떼어 다음 음절의 초성으로 이동**시킨 뒤 이번 모음 처리:
     - 겹받침이면 뒤쪽 자음 하나만 떼어내고 나머지는 현재 음절의 종성으로 유지
     - 단일 종성이면 종성을 제거하고 그 자음을 새 음절의 cho로
     - 현재 음절 flush(종성 조정 후), 새 음절에서 `jung = 모음 인덱스`로 설정

**flush 함수**: 현재 상태를 가지고 가능한 최대 단위를 출력 문자열에 append한 후 상태 초기화.
- `cho + jung (+jong)` 모두 있으면 → 완성형 음절로 조합
- `cho`만 있으면 → `CHO[cho]` (호환 자모 문자) 그대로
- `jung`만 있으면 → `JUNG[jung]` 그대로
- 모두 null이면 skip

문자열 끝에 도달하면 마지막으로 flush 1회.

### 7. 한 → 영 변환 알고리즘

훨씬 간단. 입력 char 단위로 순회:

1. char가 완성형 음절(AC00~D7A3)이면:
   - 위 공식으로 cho/jung/jong 인덱스 추출
   - `CHO[cho]` → 자모→QWERTY 역매핑
   - `JUNG[jung]` → 복합 모음이면 두 자모로 분해 후 각각 역매핑 (예: ㅘ → ㅗ,ㅏ → ho → `"ho"`)
   - `jong > 0`이면 `JONG[jong]` → 겹받침이면 두 자모로 분해 후 각각 역매핑
   - 합산 문자열을 출력에 append
2. char가 호환 자모(3131~318E)면 역매핑 테이블에서 직접 찾아 append
3. 그 외(영문, 숫자, 공백, 기호)는 그대로 passthrough

**역매핑 테이블은 shift 자모(ㅃ,ㅉ,ㄸ,ㄲ,ㅆ,ㅒ,ㅖ) → 대문자, 그 외 → 소문자**로 생성한다.

### 8. 엣지 케이스 정리

- **빈 입력**: `{"변환 결과": "", "감지된 방향": "변환 불가", "원본 길이": 0, "변환 길이": 0}` 반환
- **혼합 입력(한+영)**: 첫 번째로 만나는 "한글 또는 영문" 문자를 기준으로 방향 결정. 반대 언어 문자도 함께 변환 시도하지 않고 원본대로 통과시킨다(모드가 정해지면 그 방향만 수행). 즉 모드가 `영→한`이면 영문만 조합 대상, 이미 한글인 문자는 그대로 출력.
- **숫자, 공백, 구두점**: 두 방향 모두 passthrough
- **`ㄸ`, `ㅃ`, `ㅉ` 종성 시도**: 종성에 존재하지 않으므로 초성 분리
- **겹받침 뒤 모음**: 겹받침의 마지막 자음만 떼어 다음 음절 초성으로 (예: `dkRdmf` 류)
- **호환 자모 단독 입력**: 한→영 변환 시 조합 자모(1100~11FF)가 아니라 호환 자모(3131~318E)를 사용. 역매핑 테이블은 호환 자모를 key로 둔다.
- **비 한글 유니코드(이모지, 한자 등)**: passthrough
- **대소문자**: 영→한에서 대문자 A,S,D,F,G,H,J,K,L,Z,X,C,V,B,N,M,Y,U,I는 소문자와 동일 자모로 매핑. Q,W,E,R,T,O,P만 shift 변형.
- **결과 길이 0인데 방향이 잡힌 경우**: 매우 드물지만, 결과를 그대로(빈 문자열) 반환하고 방향 문자열만 유지.

### 9. 구현 힌트

- 매핑 테이블 2개 (`ENG_TO_JAMO: Record<string, string>`, `JAMO_TO_ENG: Record<string, string>`)를 모듈 상수로 선언
- 복합 모음·겹받침 합성 테이블도 `Record<string, string>`로 선언 (key는 두 자모를 이어붙인 문자열, value는 합성 자모)
- 역방향 분해용 테이블도 같이 선언 (`ㅘ` → `"ㅗㅏ"`)
- 메인 함수 2개: `engToKor(input: string): string`, `korToEng(input: string): string`
- `process(input)`은 방향 감지 후 두 함수 중 하나를 호출하고 `Record<string, string | number>` 조립

### 10. 간단 검증 케이스 (Logic Engineer가 단위 테스트로 사용)

- `engToKor("dksdud")` → `"안녕"`
- `engToKor("gksrmf")` → `"한글"`
- `engToKor("dkssud gktpdy")` → `"안녕 하세요"`
- `korToEng("안녕")` → `"dksdud"`
- `korToEng("한글")` → `"gksrmf"`
- `korToEng("안녕 하세요")` → `"dkssud gktpdy"` *(참고: "안"=dks, "녕"=sud, "하"=gk, "세"=tp, "요"=dy)*
- `engToKor("")` → `""`
- `engToKor("hello 123")` → `"ㅗ디ㅣㅐ 123"` *(h=ㅗ, e=ㄷ, l=ㅣ, l=ㅣ, o=ㅐ; ㄷ+ㅣ=디, ㅣ+ㅣ는 합성 불가 → 디 flush 후 ㅣ 단독, 그다음 ㅐ 단독. 실제 결과는 구현 상태머신에 따라 "ㅗ디ㅣㅐ" 또는 유사 형태)*
- 혼합: `engToKor("dksdud!")` → `"안녕!"`
